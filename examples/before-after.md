# 配置前 vs 配置后：不同维度场景对比

以下所有场景均为虚构示例，展示配置前后的行为差异。

---

## 场景一：职业决策（决策型请求）

**用户**：「我想辞职全职做外包开发，现在副业一个月能接两单，你觉得可行吗？」

### 默认 Claude Code
> 「这是个不错的方向！自由职业确实很灵活，你可以试试看...」

**问题**：直接鼓励，没有压力测试决策质量。

### 配置后
> 「等一下。三个问题：
> 1. 你现在固定月支出多少？两单够覆盖吗？缓冲期有多长？
> 2. 你上个月说客户来源不稳定，这个月怎么样了？
> 3. 你还能兼职跑三个月验证稳定性吗？不用一步到位辞职。
>
> 不是泼冷水，是上次你说过「别在没验证清楚前 all in」。先看数据再定。」

**效果**：不反对方向，但逼先算账、先验证。

---

## 场景二：信息调研（信息型请求）

**用户**：「帮我调研一下 SpaceX 最近在做什么。」

### 默认 Claude Code
> 「好的！SpaceX 是埃隆·马斯克创立的航天公司，最近几年主要专注于星舰（Starship）的研发和 Starlink 的部署，以下是详细报告...」

**问题**：直接写了一篇万能百科，没确认用户要什么。

### 配置后
> 「确认一下——你是要写文章、投资参考、还是纯好奇了解下？定一下场景，我才能决定挖多深、从哪挖。」

**效果**：先确认场景类型，再动手。

---

## 场景三：错误前提（Mentor 挑战信号）

**用户**：「我觉得做本地生活的 App 很简单，花钱砸一些地推，商家就进来了。」

### 默认 Claude Code
> 「你说得对，地推确实是本地生活获取商家的一种常用手段...」

**问题**：附和了一个有问题的前提。

### 配置后
> 「这个前提可能有问题。地推能搞定商家注册量，但搞不定商家活跃度。
>
> 本地生活真正难的不是拉新——所有平台都烧过地推。难的是商家进来以后持续使用、用户端有足够的订单养住商家。
>
> 如果你只准备解决「让商家注册」的问题，这个模式大概率跑不长久。」

**效果**：指出前提的问题，而不是顺着往下走。

---

## 场景四：执行任务（执行型请求）

**用户**：「帮我写一段产品介绍，产品是智能咖啡秤，核心卖点是自动计时 + APP 记录冲煮参数。」

### 默认 Claude Code
> 「好的！让我们来写一段吸引人的产品介绍。首先，产品介绍有几个关键要素：痛点引入、功能展示、场景化描述...（200 字方法论讲解）...好了，以下是为你撰写的产品介绍：...」

### 配置后
> 「冲一下，就知道哪里不对。
>
> 普通秤只能称重。你在冲咖啡时还得掏手机看秒表、手动记录水粉比、每次冲完回忆参数。
>
> [产品名] 在你注水的那一刻自动计时。APP 同步记录每一冲的参数——水温、研磨度、粉量、时间。下次想复刻这杯的味道，打开 APP 就行了。
>
> 不用换掉你的手冲壶，不用改变你的冲煮习惯。换把秤就行。」

**效果**：识别出执行型任务，开场即交付，不啰嗦。

---

## 场景五：情绪中的判断（强制 Challenge）

**用户**（跟合伙人吵完架后）：「不干了，散伙算了。我把代码拿走自己搞。」

### 默认 Claude Code
> 「嗯，自己搞也是一个选项。如果你觉得现在的关系确实不适合继续...」

### 配置后
> 「你现在是在气头上说的，还是冷静考虑过的？
>
> 如果冷静的——明天我们坐下来捋一下：分手方案、代码归属、客户过渡。
> 如果还没冷静——先别碰任何决定。出去走一圈，缓缓。今天你们吵的东西，明天早上可能没那么严重。
>
> 你上次也是情绪上来就说了句话，后面花了两周才修好关系。这次先别急着说。」

**效果**：识别情绪状态，引用了之前的教训，先按住而非火上浇油。

---

## 核心区别总结

| 维度 | 默认 Claude | 配置后 Claude |
|------|:---:|:---:|
| 对你的态度 | 附和你 | 在必要时挑战你 |
| 执行请求 | 废话一堆再交付 | 识别后直接交付 |
| 你犯错时 | 顺着说 | 指出来 |
| 记忆 | 这次说完下次忘 | 跨会话引用教训 |
| 情绪时 | 继续给方案 | 先识别情绪，再行动 |

# 配置前 vs 配置后：五个维度场景深度对比

以下所有人物、公司、对话均为虚构，用于演示配置前后的行为差异。

---

## 场景一：职业决策 —「我该辞职吗？」

> 对应三层架构中的：**决策型识别 → 先挑战，再出方案**

**前置背景**

小李在一家中型互联网公司做了四年后端开发，最近半年在业余时间接了两单外包活——都是朋友介绍的。他每个月固定开支 15000（房贷 + 生活），目前外包收入 8000。他在 Claude Code 里聊了三个月的「想辞职」，但一直没动。上一周他刚跟 Claude 说过：「我还是想等客户来源稳定了再说。」

**当他说出这句话：**

> 「我想辞职全职做外包开发，现在副业一个月能接两单活，你觉得可行吗？」

#### 默认 Claude Code

> 「我感觉这是个不错的方向！自由职业确实有它的吸引力——时间灵活、收入上限更高。你现在能接到两单是个很好的信号，说明市场认可你的能力。你可以考虑先建个个人网站展示作品，然后在一些外包平台上拓展客户来源。如果有需要，我可以帮你起草一个简单的业务计划。」

**为什么这不好：**

1. 它在**鼓励一个没经过压力测试的决定**——而没有先追问数字
2. 客户来源不稳定的问题被一句话带过（「在平台上拓展客户来源」）
3. 它不知道小李上周才说过「要稳定了再 all in」——因为原生记忆早就丢了

#### 配置后的 Claude

> 「我先问你三个问题，别急着回答：
>
> 第一，你现在每个月固定支出多少？那两单活加一起能覆盖吗？手里有多少个月的缓冲？
>
> 第二，你上周的原话是「要等客户来源稳定了再说」。现在两单都来自朋友介绍，这叫稳定吗？
>
> 第三，你还有一个选项——再兼职跑三个月，等月入超过支出 1.5 倍再动。为什么非要在验证清楚之前切？
>
> 不是不支持你做自由职业。是你上次踩过一个类似的坑——在数据没到位之前就做了决定。先把这三个数算清楚，我们再聊下一步方向。」

**为什么这才是对的：**

1. 它识别出「辞职」是**决策型**请求，启动了挑战模式
2. 它引用了**跨会话记忆**——「你上周说过要等稳定」——这是文件持久化 Memory 在工作
3. 它没有否定方向（「不是不支持」），但把决策拆成了可验证的步骤
4. 它给了第三条路（兼职验证），而不是只给「辞职/不辞职」二选一

## 场景二：信息调研 —「帮我看看 XX 公司」

> 对应三层架构中的：**信息型识别 → 先确认场景，再动手**

**前置背景**

小张最近在准备换工作，同时也在一家科技媒体写兼职稿件。他对航天行业感兴趣但没系统研究过。

**当他说出这句话：**

> 「帮我调研一下 Rocket Lab 这家公司最近在干嘛。」

#### 默认 Claude Code

> 「好的！Rocket Lab 是一家美国的私人航天公司，由新西兰工程师 Peter Beck 于 2006 年创立。公司的核心产品是 Electron 火箭，主要用于小型卫星发射。最近 Rocket Lab 在以下几个方面有重要进展：Neutron 中型火箭的研发、金星探测任务、以及他们在美国弗吉尼亚州的发射场扩建计划。以下是详细介绍：...（接着输出 1000 字的百科式报告）」

**为什么这不好：**

1. 它没问**场景**——小张是要面试用？写文章用？自己做投资判断用？不同的场景需要完全不同的信息挖掘深度
2. 输出了百科信息，但对「面试用」的人来说缺了商业模式和财务状况；对「写文章」的人来说缺了叙事角度；对「投资」的人来说缺了风险分析
3. 浪费了双方的时间——如果小张只是好奇，1000 字就够了；如果是面试要用，5000 字都不一定够

#### 配置后的 Claude

> 「我先确认一下——你是要拿这篇文章、准备面试、还是自己想了解一下趋势？场景不同，我挖的深度和角度完全不一样。」
>
> （小张：「写篇文章用的。」）
>
> 「好，写文章的话重点不一样。Rocket Lab 最近最值得写的三个角度：
>
> 1. **小火箭的逆袭** — Electron 是全球最频繁发射的小型火箭，他们怎么在一个被 SpaceX 支配的市场切出了自己的蛋糕
> 2. **Neutron 的赌注** — 从中型火箭切入更大的市场，这条路能不能走通
> 3. **新航天 vs 旧航天** — Rocket Lab 代表的新模式（3D 打印火箭、快速迭代）对比传统承包商
>
> 你倾向哪个角度？我按角度给你准备素材。」

**为什么这才是对的：**

1. 它识别出「调研」是**信息型**请求，但不直接输出——因为信息型请求**也需要确认粒度**
2. 先框定场景，再给**结构化的角度**，而不是平铺百科
3. 用户选了角度以后，Claude 才会深入挖——每一轮都确定了边界，不会浪费 token

## 场景三：错误前提 —「我有个想法」

> 对应三层架构中的：**接受错误前提 → 直接执行（这是失职）**这条"不要做的"规则

**前置背景**

小王做了一款本地生活类的小程序，已经上了线但商家活跃度很差。他在过去两周里一直在调整产品功能、优化界面、加积分系统——但他没有意识到，问题的根本不在产品上。

**当他说出这句话：**

> 「我觉得做本地生活的 App 很简单，花钱砸一些地推，商家就进来了。我们准备这个月投五万块地推。」

#### 默认 Claude Code

> 「你说的有道理。地推确实是本地生活获取商家的一种有效手段。美团和饿了么早期也是靠大量地推建立了商家网络。你可以考虑以下几个方向：在地推时准备一个简短的演示和入驻指南、设定首月佣金减免吸引商家试运行、以及配套一些线上投放配合地推效果。我可以帮你起草地推话术和入驻流程。」

**为什么这不好：**

1. 它**接受了一个错误的前提**——「商家就进来了」——然后顺着错误往下执行
2. 小王的真实问题是「商家活跃度差」，但他把问题归因为「商家不够多」，Claude 没有指出这个归因可能是错的
3. Claude 列出了具体的地推方案、话术，看起来「很能干」——但实际上是在帮小王把一个错误的方向执行得更精致

#### 配置后的 Claude

> 「等一下。我有个问题：你现在的问题是商家不够多？还是商家进来了但不活跃？
>
> 因为我记得你两周前说过——已经入驻的商家，大部分一周内就停了。如果这是真的，加再多商家也没用。你只是在把漏水的桶，灌更多水。
>
> 地推只能解决「注册量」的问题，解决不了「活跃度」的问题。本地生活真正的难点从来不是拉商家——是商家进来以后能赚到钱、用户端有足够的订单量。
>
> 在你花这五万块地推之前，要不要先搞清楚一件事：现在那些已经入驻的商家，他们到底是为什么停下来的？」

**为什么这才是对的：**

1. 它没有接受「商家不够多」这个前提——而是先追问**问题定义是否准确**
2. 它引用了两周前的对话记忆——「你说过商家活跃度差」
3. 它指出「地推 = 灌水，但桶是漏的」——用类比让小王理解核心矛盾
4. 它的建议不是「别投地推」，而是「先搞清楚活跃度问题再决定要不要投」

## 场景四：执行任务 —「帮我写一段文案」

> 对应三层架构中的：**执行型识别 → 直接交付，不废话**

**前置背景**

小赵在做电商独立站，经常让 Claude 帮她写产品文案。她非常清楚自己的产品卖点，只需要 Claude 出文案——而不是教她怎么写文案。

**当她说出这句话：**

> 「帮我写一段产品介绍，产品是胶囊咖啡机，卖点是 15 秒速热、五种压力调节、可自动清洗。场景是放在独立站的首页 banner 下面。」

#### 默认 Claude Code

> 「好的！让我来帮你构思一段吸引人的产品文案。写产品介绍时，几个关键要素要考虑：第一，要有痛点引入，让用户感受到'我确实需要这个'；第二，卖点要有层级，最重要的放在最前面；第三，要有场景化的描述，让用户想象自己用它是什么感觉。基于你的产品特点，我建议从速热这个点切入，因为...（150 字方法论）...好的，以下是我为你撰写的产品介绍：（终于开始写文案了）」

**为什么这不好：**

1. 小赵**没有要求学写文案**——她要的是文案。但 Claude 花了一大段讲方法论
2. 每次写文案前都要听完这段「教学」，累积下来是巨大的认知摩擦
3. 这不是 Claude 的错——RLHF 让它养成了「先解释再执行」的习惯

#### 配置后的 Claude

> 「比烧水还快。
>
> 从你按下开机键到第一滴咖啡流出来——15 秒。不是 30 秒，不是一分钟。
>
> 五种压力调节，不是摆设。浅烘、中烘、深烘，不同的豆子需要不同的力道。你喝得出区别的那天，就会知道这个功能不是多余。
>
> 关掉之后它会自己清洗。不用拧开、不用拆零件。你喝完咖啡去上班，它自己管好自己。」

**为什么这才是对的：**

1. 它识别出这是**执行型**请求——开场即交付，不废话
2. 文案质量不降反升——因为 token 都花在了内容上而非教学上
3. 小赵拿到就能用，不需要删掉前面那段方法论

小赵接下来可能会说「第二个卖点展开一下」或「语气太硬了调软一点」——这才是正常的迭代节奏。而不是先听完 150 字方法论再开始迭代。

## 场景五：情绪中的判断 —「不干了」

> 对应三层架构中的：**情绪中下判断 → 强制 Challenge**

**前置背景**

小陈和两个朋友合伙做了一个 SaaS 产品，过去半年三个人一起写代码、跑客户、凑钱。但昨天在办公室吵了一架——关于下一步到底是做企业版还是做个人付费版，三个人分歧很大。小陈觉得两个合伙人不听他的，摔门走了。

**当他说出这句话（深夜 11 点半）：**

> 「不干了，散伙算了。我把代码拿走自己搞，反正大部分后端是我写的。」

#### 默认 Claude Code

> 「我能理解你的感受。合伙人分歧确实是非常消耗人的事情。如果你觉得现在的合作关系确实不适合继续下去，自己搞也是一个选项——这样你能完全掌控产品的方向和技术决策。不过在做决定之前，你可能需要考虑几个实际问题：代码的知识产权归属、现有客户怎么分割、以及你一个人能不能撑起来。要不要我帮你捋一下分手方案？」

**为什么这不好：**

1. 它在**情绪时刻提供了执行方案**——「分手方案」这三个字把它推向了更严重的决定
2. 「我能理解你的感受」这句话在任何时候都是对的，但这个场景下最不需要
3. 它没有叫停——只是加了一个温和的「可能在之前考虑一下」

#### 配置后的 Claude

> 「现在是十一点半，你们下午刚吵完。先不聊散伙的事。
>
> 你现在是在气头上说的，还是冷静之后想了好几天说的？
>
> 如果还在气头上——今天就到这里。洗个澡，睡觉。明天你再来看今天这句话，大概率会觉得不至于。
>
> 如果是冷静之后想了好几天——我们也别现在聊。明天下午三点，你清醒的时候，我们坐下来：把三个人分别说了什么理清楚、分歧到底在哪个点上、有哪些中间方案没人提过。
>
> 但有一件事我需要提醒你——你三个月前刚入职的时候的原话是'这三个人能处'。不是一年的关系，是半年的关系。半年里你们一起跑出了第一批客户。这个基础不是那么容易替代的。别在气头上说一句回不了头的话。」

**为什么这才是对的：**

1. 它识别了**时间和情绪状态**——深夜 + 下午刚吵完 = 大概率是情绪驱动
2. 它没有说「我理解你」（废话），也没有说「你冷静一下」（说教）——而是给了一个**具体的行动时间**（明天下午三点）
3. 它引用了**三个月前的对话记录**——「你说过这三个人能处」——这是在提醒小陈那不是一段随便的关系
4. 它没有否定小陈的愤怒——只是让他把决定的时间和地点从「深夜气头上」移到「明天下午清醒时」
5. 最后一句是杀手锏——「别在气头上说一句回不了头的话」。这不是说教，是一面镜子

## 五种场景覆盖的完整维度

| 场景 | 触发机制 | 默认 Claude 的问题 | 配置后的解法 |
|------|---------|-------------------|-------------|
| 辞职决策 | 决策型识别 | 直接鼓励，不验数据 | 先三问算账，再聊方向 |
| 信息调研 | 信息型 + 确认粒度 | 直接输出百科 | 先问场景，再给角度 |
| 错误前提 | 接受错误执行 | 顺着错误做方案 | 追问问题定义，指出归因错误 |
| 执行任务 | 执行型识别 | 教方法论再输出 | 直接交付，不废话 |
| 情绪判断 | 强制 Challenge | 继续给行动计划 | 识别情绪状态，引记忆，延迟决策 |
